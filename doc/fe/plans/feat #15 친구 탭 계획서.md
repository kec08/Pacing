## 1. 목적 및 배경

Pacing v1.2 기획서에서 추가된 5탭 구조 중 `친구` 탭을 구현한다. 사용자는 닉네임 검색, 추천 친구 확인, 친구 요청 수락/거절, 친구 목록 관리를 통해 러닝 중 같이 듣기와 플레이리스트 공유로 이어지는 기본 관계망을 만들 수 있어야 한다.

이번 범위는 친구 기능의 MVP로 잡는다. 연락처 권한과 전화번호 해시 매칭은 구조를 열어두되, 실제 연락처 기반 추천은 별도 후속 작업으로 분리한다.

## 2. 사용자 시나리오

```
사용자가 하단 친구 탭에 진입하면
→ 내 친구 목록과 추천 러너를 확인하고
→ 상단 친구 추가 아이콘에서 받은 친구 요청을 풀스크린으로 확인하며
→ 닉네임 검색 또는 추천 카드에서 친구 요청을 보낸다
→ 상대가 수락하면 내 친구 목록에 표시된다
```

## 3. 화면 구성

### 주요 화면
- 친구 탭 메인: 검색바, 상단 친구 추가 아이콘 버튼, 내 친구 목록, 추천 친구 섹션
- 친구 요청 풀스크린: 요청자 아바타/닉네임, 수락/거절 버튼, 닫기 버튼
- 친구 카드: 아바타/닉네임, 최근 러닝 상태, 같이 듣기 연결 진입 여지를 둔 UI
- 추천 친구 카드: 추천 타입, 닉네임, 친구 추가 버튼

### UI 요소
- 검색바: 닉네임 기반 유저 검색
- 섹션 헤더: 내 친구, 추천 친구
- 빈 상태: 요청/친구/추천이 없을 때 안내
- 로딩/에러 상태: Firestore 조회 실패 또는 권한 미설정 상태 표시

## 4. 데이터 흐름

### 사용 데이터
- Firestore 컬렉션:
  - `users/{uid}`: 닉네임, 프로필 이미지, 최근 상태 조회
  - `users/{uid}/friends/{friendUID}`: 내 친구 목록
  - `friendRequests/{requestID}`: 친구 요청 상태 관리
- Realtime DB 경로:
  - `activeRunners/{uid}`: 가까운 러너 추천 후보로 활용 가능
- Contacts:
  - 이번 MVP에서는 권한 요청 및 해시 매칭 미구현
  - 후속 작업에서 `phoneHash` 기반 추천 추가

### 상태 관리
- ViewModel:
  - `FriendsViewModel`
- Published 프로퍼티:
  - `friends: [FriendUser]`
  - `incomingRequests: [FriendRequest]`
  - `recommendedUsers: [FriendUser]`
  - `searchText: String`
  - `searchResults: [FriendUser]`
  - `isLoading: Bool`
  - `errorMessage: String?`

## 8. 특이 사항 / 기술 검토

- `origin/dev`에는 #14 네이버 로그인까지 머지되어 있으며, `feat/15-friends-tab`은 최신 `dev` 기준으로 분기했다.
- Firestore 쿼리는 닉네임 prefix 검색 제약이 있으므로 MVP에서는 `order(by: "nickname").start(at:).end(at:)` 방식 또는 제한된 전체 후보 조회 중 하나를 선택한다.
- 친구 추천 우선순위 중 연락처 기반 매칭은 개인정보/권한/해시 저장 정책이 필요하므로 별도 `feat #16` 또는 후속 이슈로 분리하는 것이 안전하다.
- 공유 탭은 이번 범위에서 placeholder만 추가하고 실제 플레이리스트 공유 기능은 별도 계획서로 진행한다.
